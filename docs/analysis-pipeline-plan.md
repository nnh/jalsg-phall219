# 解析パイプライン計画

随時更新（2026-07-21 時点）。データロック後に「固定データ → 解析データセット → 図表 → 研究者への納品物」までをClaude主導で回すための理想フロー、必要な材料、明子先生の介入点、パート分割とモデル配分、内部検証の設計をまとめる。目的は、明子先生の判断を要する要所を明確に切り出しつつ、機械的作業をClaudeが最高品質で一気通貫実行できる状態を作ること。

関連：解析ロジック・落とし穴の正本は [SAS_ClinicalTrial_ClaudeCode_共通ナレッジ.md](SAS_ClinicalTrial_ClaudeCode_共通ナレッジ.md)、ARS/ARDの調査は [cdisc-ars.md](cdisc-ars.md)、現状の到達点は [state-survey.md](state-survey.md)、SAP論点は [sap-open-issues.md](sap-open-issues.md)。

## 解析と納品の方針

解析は**SASとRの二重コーディング**で行う。SASは明子先生が使い慣れているため実務の主軸になり、Rは研究者への納品と拡張性を担う。両系統とも本解析であり、どちらか一方の検算にとどまるものではない。2系統の結果は**ARSベースでcompare**して整合を確認する。

一方、**納品はRと図表（TLF）に限る**。JALSGでは引き渡し相手が研究者で、研究者はSASを持たない（高額）。Rで納品すれば受領後に追加解析を自ら回せる。**SASプログラム・二重コーディングの事実・compare結果は納品物に含めない**（内部に留める）。二重コーディングやcompareを外に出すと、かえって不一致の解釈を巡るツッコミどころを与えかねないため、体制が確実になるまでは外に出さない。将来、体制が固まれば位置づけを見直す。

compareの粒度は、レイアウト済みのTLF（Table/Listing/Figure）レベルではなく、**CDISC ARSが提唱するARD（Analysis Results Dataset）のレベル**でそろえる。TLFは書式・結合順・セル位置の差に埋もれて機械照合しにくいが、ARDは結果値の縦持ちなので結果データフレーム同士を直接突合できる（詳細は [cdisc-ars.md](cdisc-ars.md)）。

## ゴール

研究者に渡す成果物は「SAPの図表案（第5章）に沿った番号付き図表一式（RTF/PDF）＋それを再生成・拡張できるRスクリプトとARD」。品質は、同一仕様からSASとRで独立に作った2系統のARDをClaudeが内部でcompareして担保する（compare結果もSASコードも自分用、研究者・PIには出さない）。研究者は納品されたRコードとARDから、追加解析や別表示を自分で作れる。

## 明子先生の介入ポイント

人の介入は**必須2点＋条件付き2点**に絞る。必須以外はClaudeが実行し、条件が発生したときだけ止まる。

必須（必ず止まる）：

1. **データロックの実施**：ロックはDM/統計責任者の権限行為。Claudeは実行しない。パイプラインの起点。
2. **最終図表の学術的レビューと納品承認**（承認ゲート②）：数値だけでなく臨床的・統計的に妥当な表現かの確認と、研究者へ出す責任判断。ここだけが完成物に対する唯一の必須ゲート。

条件付き（該当時のみ止まる）：

3. **新規解釈が生じた時のARS仕様確認**（承認ゲート①）：SAPは2026-07-21固定済みで、ARS仕様は原則その機械的な書き起こし。**SAP/specで未確定の解釈判断が新たに出た時だけ**その点に限って明子先生に確認する（CHR/CHRp/LFS階層・同日tie-break・RFS対象集団など既決事項は確認不要）。新規判断がなければ止まらない。
4. **compare不一致のアジュディケーション**：R系ARDとSAS系ARD、または予測検算値が食い違った時のみ、Claudeが原因を切り分けて両案を提示し、最終採否を明子先生が判断する。差分ゼロなら提示も停止も不要。

## 一気通貫か分割か

理想は「データロック後、ClaudeがPhase 1→6を進め、内部compareで差分ゼロを確認して承認ゲート②に納品物を載せる」こと。通常止まるのは**🔁 モデル切替2回（Phase 1でFableへ／Phase 3でSonnetへ）と、🧑 必須ゲート②（納品承認）だけ**。ARS仕様の新規解釈（ゲート①）とcompare不一致（4-2）は⚠️条件付きで、該当したときのみ止まる。前半Fable・後半Sonnetの2ブロックにまとめることで切替頻度と手戻りを抑える。

作業単位（サブ番号）と推奨モデル。原則：**精度・判断が要る前半（ARS設計・ADaM・中核エンドポイント導出）はFableで通し、反復的で大量出力になる後半（ARD生成・図表描画）はSonnet 5**。モデルは前半Fable→後半Sonnetの**2ブロックにまとめ、切替は原則1回だけ**。中核エンドポイント導出のためにFableが要る一方、その前後のADaM実装も同ブロックに含めれば切替が消え、トークン増もADaMは一度きりで限定的。反復回数が多くトークンが膨らむARD生成・図表描画のみSonnetに落とす。

停止列：🔁＝ここで`/model`切替（全体で1回）。🧑＝人（明子先生）の介入・必須。⚠️＝条件付き（該当時のみ停止）。空欄＝前行と同じモデルで継続。

| Phase | 作業単位 | モデル | 停止 | 理由 |
|---|---|---|---|---|
| 1 | ARS仕様インデックス化・spec突合・材料チェックリスト作成 | Fable | 🔁 | SAP解釈と設計判断が全下流を規定。ここでFableに切替 |
| 2-1 | R系 ADaM builder（{admiral}系）実装 | Fable | | 中核導出と同ブロックにして切替を回避。ADaMは一度きりでトークン限定的 |
| 2-2 | **中核エンドポイント導出（EFS/OS/RFS：tie-break・CHR/CHRp/LFS階層・起算日）** | **Fable** | | 最高精度が要る中核。誤ると全図表に伝播 |
| 2-3 | SAS系 ADaM builder（既存CSVtoSASDS.sas改修：gml219→phall219置換、不要ドメイン除去） | Fable | | 同上。一度きりの変換 |
| 3-1 | R系ARD生成（{cards}/{cardx}） | Sonnet 5 | 🔁 | 解析数だけ反復しトークンが膨らむ。ここでSonnetに切替 |
| 3-2 | SAS系ARD生成（同一ARS仕様で独立実装） | Sonnet 5 | | 独立実装だが定型・反復 |
| 4-1 | ARDレベルcompareハーネス実装 | Sonnet 5 | | 決定論的 |
| 4-2 | 不一致の原因切り分け・アジュディケーション補助 | Fable | ⚠️🧑 | 差分が出たときのみFableに一時切替＋明子先生判断。差分ゼロなら停止なし |
| 5 | 図表描画・番号付け・Rスクリプト同梱・結果サマリ | Sonnet 5 | | 反復的な定型整形 |
| 6 | 反省録・共通ナレッジ更新 | Sonnet 5 | | 記録 |

運用：**通常の停止は🔁が2回（Phase 1でFableへ、Phase 3でSonnetへ）と🧑の必須ゲートのみ**。⚠️（ゲート①のうち新規解釈が生じた時、4-2で差分が出た時）は該当したときだけ止まる。印のない行は前行と同じモデルで連続実行する。

## 内部検証とARDレベルcompare

SASとRはいずれも本解析だが、両者を突き合わせるcompare自体は納品物ではなく**Claude自身の内部検証**。同一の固定データ・同一のARS仕様から、R系とSAS系を互いに参照せず独立実装し、**ARD（結果値の縦持ち）のレベルで突合**する。TLFレベルではなくARDレベルにするのは、書式・結合順・セル位置の差に埋もれず結果値そのものを機械照合できるため（[cdisc-ars.md](cdisc-ars.md)）。

### 独立性の担保

- R系（Claudeセッションα）とSAS系（Claudeセッションβ）は、互いのコードを見ずに同じspec（ARS仕様・エンドポイント仕様）だけを入力にする。共有するのは仕様と固定データのみ。二重コーディングの独立性がcompareの検出力を担保する。
- ADaMも両系統で独立に作るのが理想だが、コスト次第では「主要エンドポイントのADaMだけ独立導出、記述統計は共有ADaMから両系統集計」の2段構えにする。

### compareの単位

1. **ADaMレベル**：`adtte`（EFS/OS/RFS）・`adsl`（母集団フラグ）・`adrs`（CHR/CMR）などを両系統でキー結合し、AVAL・フラグ・イベント日を数値許容差（例 日数は完全一致、割合は1e-9）で突合。
2. **ARDレベル**：各解析のARD（`variable`/`stat_name`/`ResultGroup`をキー、`stat`が値）を両系統でそろえ、`waldo::compare()` 等で結果値を突合。これが検証の主軸。
3. **予測検算値との突合**：主要エンドポイントは`efs_subject_check`の手計算値とも突合（共通ナレッジ§10）。ARD・予測値・両系統の三者一致で確度が最も高い。

### 実装と出力

- 検証系SAS内のSAS×SAS検証は`proc compare`。R↔SAS はARDを共通中間形式（縦持ちCSV/parquet）にそろえて突合するハーネスを`scripts/`に置く。
- 出力は `compare/reconciliation_YYYYMMDD.md`（自分用）：解析ごとに「一致／不一致（該当キー・R値・SAS値・予測値・差分）」を一覧化。**差分ゼロなら記録して先へ進む。差分ありのときだけ原因分類（丸め・母集団定義差・欠測扱い・ロジック差）を付けて明子先生に提示**する。
- `compare/` は`.gitignore`対象（被験者値を含むため）。Boxに保管する。PIには出さない。

## 指示に必要な事項

Claudeへ各Phaseを依頼するときのプロンプトに、最低限含める要素。これがそろっていれば手戻りが最小になる。

- **対象Phaseと作業単位・系統**（例：「Phase 3 R系、図表5.4.7.1〜5.4.7.5のAE頻度表のARDを{cards}で」）。
- **入力の所在と版**：固定データのパス・納品日・症例数、参照するspec/ARS仕様のファイルとバージョン、`saihi.csv`の版。
- **母集団と分母の定義**：FAS/SAF/PPSのどれか（AnalysisSet）、サブグループ（PN導入例・移植例）の識別条件。
- **エンドポイント解釈の参照先**：`efs_plan_vX.Y`のどの規定に従うか、tie-break・起算日・階層判定の明示。
- **ARS/出力仕様**：解析名・変数・統計手法（Operation）・群（ResultGroup）と、図表番号・タイトル・レイアウト・出力形式（RTF/PDF）・保存先。
- **QC基準**：突合相手（検証系ARD／予測検算値）と許容差、compareレポートの出力先（自分用）。
- **禁止・制約事項**：SASファイルはCP932・Edit禁止・PowerShell経由、生データ/出力/ログ/ARD/compareはgit禁止・Box保管、被験者データの中身をチャットに出さない、compareは内部検証でPI非提示。
- **独立性の指示**（検証系との二重実装時）：「相手系統のコードを参照せず、ARS仕様と固定データのみから実装せよ」。
- **完了条件と次アクション**：何をもって完了とし、どのゲートに何を提示するか。

### 着手前チェック

各Phase・各作業単位に着手する前に、上記「指示に必要な事項」を**プリフライト・ゲート**として全項目チェックする。**1項目でも欠けていれば着手しない**。これは推奨ではなく必須の停止条件とする。

- 全項目が揃っている → 着手する。
- 欠けている項目がある → **先に進めず**、欠落項目を箇条書きで明示し、明子先生／齋藤に提示して充足を待つ。欠落を推測・既定値・前回値で埋めて進めることは禁止（誤った前提が全下流に伝播するため）。
- 「停止」列に🔁がある行はモデル切替の完了を、🧑がある行は人の承認を、⚠️は条件の非該当（または該当なら対応完了）をチェック項目に含める。未了なら同様に着手しない。
- 実装上の担保：各Phase冒頭で、揃っている項目・欠けている項目を一覧化した短いプリフライト表を出力してから着手する。欠落があればそこで停止する。

## リスクと落とし穴

- **データロック未確定のまま着手しない**：ロック前データで作った図表は作り直しになる。ロック版であることの明示確認を材料チェックの必須項目にする。
- **`saihi.csv`未反映**：母集団が決まらないと全図表の分母が動く。ロック直後の最優先タスク。
- **疾患別ロジックの誤流用**：AML（GML219）のOVRLRESP/CRロジックをALLのHEMARESP/CHR階層に流用しない（共通ナレッジ§5.5）。SAS系が流用する既存CSVtoSASDS.sasがGML219流用のままである点は改修時の最大の対象。
- **FAとAEの取り違え**：治療相別AE GradeはFA、SAEはAE（共通ナレッジ§11）。
- **compareの見かけ上の一致**：母集団や欠測扱いが両系統で同じ「間違い方」をすると一致してしまう。独立実装の徹底と、予測検算値との三者突合で担保する。
- **Rの再現性**：研究者が受領後に再実行できるよう`renv`でパッケージ版を固定する。固定しないと手元で動かず「追加解析できる」という納品の狙いが崩れる。
- **SAS・compareを外に出さない**：SASプログラム・二重コーディングの事実・compare結果は非納品。不一致の解釈を巡る不要な議論を避けるため、体制が固まるまでは内部にとどめる。
- **スケジュール**：CRBスケジュール（変更審査2026-09-17〈8/20締切〉、終了報告2026-10-15〈9/17締切〉）に対し、データロック後の解析期間を逆算する。遅延有無はPI/DMに確認。
